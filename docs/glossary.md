# xxskill 术语表

这份术语表给初学者快速查阅，遇到不懂的词来这里找。术语按主题分组，每条用一句话解释，尽量说人话。

---

## 一、产品方法论（想清楚）

**北极星指标** — 最能反映你给用户创造了多少核心价值的那一个数字，比"注册量"这种虚荣指标更值得盯。

**精益画布** — 把商业模式拆成问题、客户、价值主张等 9 个模块的一页纸工具，每个模块都是待验证的假设而非事实。

**MVP** — 最小可行产品，只做能验证核心假设的最少功能集，不是把完整产品砍一半。

**JTBD** — Jobs-to-be-Done 理论，核心思想是"用户不是买产品，是雇产品来完成任务"。

**MoSCoW** — 功能优先级分类法：Must 必须有 / Should 应该有 / Could 可以有 / Won't 不会有。

**虚荣指标** — 看着漂亮但不能指导决策的数字，如注册数、浏览量；对应的才是北极星指标。

**AARRR** — 海盗指标，把用户生命周期拆成获取、激活、留存、推荐、收益 5 个环节逐个检查。

**HEART 框架** — Google 提出的用户体验度量框架：满意度、参与度、采用率、留存率、任务成功。

**UVP** — 独特价值主张，一句话说清"你做什么 + 为谁做 + 为什么选你而不是现有方案"。

**CAC / LTV** — 获客成本与用户终身价值，LTV/CAC 大于 3 通常算健康商业模式。

**早期采用者** — 最痛、最急、最愿意尝试不完美方案的第一批用户，不是泛泛的"目标市场"。

**最危险假设** — 一旦错了整个产品就没存在意义的那个假设，应优先用最小成本验证。

**RICE** — 需求优先级量化打分：得分 = (Reach 影响人数 × Impact 影响程度 × Confidence 信心) ÷ Effort 工作量，分数越高越优先做。

**Problem Statement** — 问题陈述，一句话定义"谁的什么痛"，是需求阶段通向方案阶段的枢纽，避免直接跳到功能。

**Mom Test** — 用户访谈方法（出自《The Mom Test》）：问用户过去的具体行为，不问未来的意愿。因为问"你会用吗"妈妈都会说"会"骗你，问"上次你怎么办的"才说真话。

## 二、AI 能力与数据评估

**Data-Centric AI** — Andrew Ng 提出的思想：固定模型、迭代数据，往往比固定数据、换模型效果更好。

**评估集** — 一组有标准答案的测试样本，用来量化 AI 输出质量，相当于给 AI 出的考试卷。

**离线评估** — 上线前用评估集跑分，看准确率等指标；注意离线指标好不等于上线效果好。

**在线评估** — 上线后看真实用户行为（完成率、重试率、负反馈），弥补离线评估和真实体验的 gap。

**幻觉** — AI 输出"看起来对但其实错了"的内容，是 AI 产品区别于传统软件的主要失败模式。

**token 成本** — 调用大模型按用量计费的费用，每次请求都要钱，需做预算和限流控制。

**few-shot 示例** — 在 prompt 里给 AI 看几个输入输出样例，帮助它理解任务格式，注意不能和评估集重复。

**prompt 工程** — 通过设计角色、能力、知识库、输出格式、示例来引导 AI 输出符合预期的结果。

**prompt 注入** — 攻击者在用户输入里夹带指令，试图让 AI 偏离原定角色或泄露敏感信息的安全风险。

**Agentic 模式** — AI 自己规划子任务、调用工具、反思重试的多轮模式，适合单轮 prompt 搞不定的复杂任务。

**流式响应** — AI 边生成边返回片段，前端逐字渲染，比等全部生成完再返回体验更好。

**降级容错** — AI 超时或报错时切换到本地规则或兜底回复，保证用户体验不中断。

**function calling** — 让模型按约定格式返回要调用的函数名和参数，是实现 Agent 工具调用的基础。

## 三、微信小程序基础设施

**云开发** — 微信小程序官方的 Serverless 后端，提供云函数、数据库、存储，免自己搭服务器。

**云函数** — 部署在云端的 Node.js 函数，处理数据校验、存储、权限等后端逻辑，按调用量计费。

**ICP 备案** — 工信部要求的互联网信息服务备案，小程序必须备案通过才能上架，个人或企业主体均可。

**UGC** — 用户生成内容，指用户发布且其他用户可见的内容（如评论、社区），备案时需额外声明审核机制。

**隐私保护指引** — 小程序后台填写的用户信息收集声明，只勾选实际用到的 API 对应信息项，多勾会被审核打回。

**AppID** — 小程序的唯一身份标识，在微信公众平台注册后获得，需填入 project.config.json。

**API 代理** — 通过自己的服务端中转调用第三方 API 的做法，可隐藏密钥、做限流和缓存，避免前端直连暴露 key。

**key 管理** — 对 API 密钥统一存储、按环境隔离、不硬编码进代码、不提交到仓库的安全实践。

**openid** — 微信给每个小程序用户的唯一标识，不随用户换设备变化，用于识别"是不是同一个人"。不是微信号，不能用来主动联系用户。

## 四、设计系统与组件

**设计 Token** — 把颜色、字号、间距、圆角等设计值抽成变量，统一管理、一处修改全局生效。

**rpx** — 小程序响应式像素单位，750rpx 等于屏幕宽度，可自动适配不同屏幕尺寸。

**WCAG AA** — Web 内容无障碍标准的中级合规等级，要求文字与背景对比度不低于 4.5:1。

**Atomic Design** — 原子化设计方法论，把界面拆成原子→分子→组织→模板→页面分层构建。

**骨架屏** — 页面加载时用灰色色块占位的组件，比空白页或转圈更能降低用户等待焦虑。

**防抖 / 节流** — 控制高频事件触发频率的两种手段：防抖只执行最后一次，节流按固定间隔执行。

## 五、数据埋点与监控

**WeAnalysis** — 微信官方的数据分析平台（We 分析），支持事件上报、漏斗分析、留存分析。

**wx.reportEvent** — 小程序前端上报自定义事件的 API，所有属性值必须转成字符串。

**元事件** — We 分析后台注册的事件定义，事件 ID 大小写敏感、发布后不可修改。

**T+1** — 数据次日才出现的延迟模式，今天发布的小程序明天才能在后台看到数据，别当天反复查。

**转化漏斗** — 把用户从进入到完成目标的关键步骤串成漏斗，监控每步转化率以定位流失节点。

**数据漂移** — 用户输入的分布发生了变化（如提问类型变了），导致 AI 在新数据上表现下降。

**概念漂移** — 用户对"好结果"的期望随时间变了，原来的合格标准不再适用。

**静默失败** — AI 还在正常返回但质量已差，用户却无感知继续使用，是 AI 产品最大的隐性坑。

## 六、迭代与性能优化

**P0/P1/P2/P3 分级** — 问题严重程度分级：P0 阻断核心路径必须修、P1 重要建议修、P2 工程化可延迟、P3 极致优化。

**benchmarkLevel** — 微信给出的设备性能评分（数值越小性能越差），用于判断是否触发低端机降级策略。

**setData** — 小程序更新页面数据并触发渲染的 API，频繁或全量调用会导致卡顿，需做阈值和批量优化。

**分包加载** — 把小程序拆成多个子包按需下载，减小主包体积、加快首屏启动。

**cover-view / cover-image** — 能覆盖在 camera 等原生组件之上的特殊组件，普通 view 无法盖住原生组件。

**DPR** — 设备像素比（物理像素 / 逻辑像素），Canvas 绘制时需按 DPR 放大才能在高清屏不模糊。

**Core Web Vitals** — Google 定义的网络体验核心指标（LCP/CLS/INP），出海网站性能优化的基准。

## 七、平台架构

**SHARED/wechat/web 分层结构** — 本技能包的平台特化组织方式：SHARED.md 写跨平台通用原则，wechat/SKILL.md 写微信实现，web/SKILL.md 写出海网站实现（待补充）。

**AGENTS.md** — OpenAI 2025 年联合 Google、Cursor 等推出的开放标准，用一份 Markdown 给 AI 编码代理当"项目入职手册"，目标是统一各厂商私有的配置文件（如 CLAUDE.md、.cursorrules）。

**QPS** — Queries Per Second，每秒请求数，衡量接口压力的指标。QPS 100 表示每秒 100 次调用，超过接口限额会被拒。

**xxskill** — 本技能包的主入口，双模式路由器：任务前路由（你的想法该用哪个 skill）+ 任务后导航（刚做完一个 skill，下一步该干什么）。

## 八、需求澄清方法论（xx-clarify 专用）

**outcome statement** — 用 JTBD 表达需求的句式："当 [情境]，我想 [做什么]，以便 [达成什么结果]"，避免直接跳到功能词。

**Forces of Progress** — JTBD 的核心工具，分析用户切换产品时的四种力：推动力（push）、吸引力（pull）、焦虑（anxiety）、惯性（inertia）。

**walking skeleton** — 用户故事映射里的"行走的骨骼"，由一级用户任务组成的用户旅程主线，画全它就能暴露哪些环节没人想、哪些断掉了。

**Kano 模型** — 狩野纪昭提出的需求分类法，把需求分成基本（没有就怒）、期望（越多越满意）、兴奋（没有不怨有了惊喜）、无差异、反向五类，用于澄清时防止把兴奋当基本。

**OST（Opportunity Solution Tree）** — Teresa Torres 提出的结构化工具，四层：Outcome（业务结果）→ Opportunity（用户机会/痛点）→ Solution（解决方案）→ Experiment（实验），强制区分"机会"和"方案"。

**Switch Interview** — JTBD 的回溯访谈法，让用户回忆上次切换产品的完整时间线，挖出用户自己都说不清的焦虑和惯性。

**澄清卡** — xx-clarify 的输出物，包含 problem statement + 机会清单 + 假设清单 + 建议下一步，可直接喂给 xx-research 验证或 xx-prd 定义 MVP。
