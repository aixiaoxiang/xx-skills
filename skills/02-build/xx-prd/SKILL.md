# 产品设计

> 先想清楚用户要完成什么任务，再决定做什么功能。不是反过来。

## 这个技能做什么

帮你在30分钟内完成从"我有商业模式"到"我有一份可执行的一页PRD"。输出一份**问题优先的一页PRD**。

大多数PRD的问题：一上来就写功能列表。好的PRD：先写用户要完成的任务和要解决的问题，功能只是解决问题的手段。

**读者说明：** 本 skill 主要给 AI 执行规范（按模板生成 PRD、不跳步），次要给人判断标准（拿到 PRD 能看出哪里不对）。用户不需要自己写，只需要会用自然语言提需求、能看出 PRD 哪里不对并反馈。

---

## 四步流程

### 第一步：JTBD 任务定义（10分钟）

Jobs-to-be-Done 理论的核心：**用户不是买产品，是雇产品来完成任务。**

"人们不想要1/4英寸的钻头，他们想要1/4英寸的洞。" — Theodore Levitt

**AI 参考输出格式（JTBD 任务句）：**

```
当 [场景] 时，
我想要 [完成任务]，
以便于 [达成目标]。
```

**任务有三层（以小象取色为例）：**

| 层次 | 问题 | 示例（小象取色） |
|------|------|------------------|
| **功能任务** | 用户要完成什么操作？ | 取到一个颜色的准确色值 |
| **情感任务** | 用户想获得什么感受？ | 不靠肉眼猜色、获得专业感 |
| **社会任务** | 用户想被别人怎么看？ | 被认为懂配色、有品味 |

**判断标准（给人）：**
- 任务描述里有没有出现你的产品名或功能名？→ 不应该有
- 任务会不会因为技术变化而消失？→ 不会的才是真需求
- 同一个任务，现在用什么产品完成？→ 你的竞品不一定是同类产品

**AI 执行约束：**
- 任务句必须按"当…时，我想要…，以便于…"三段式输出
- 三层任务必须都写，不能只写功能任务
- 任务描述禁止出现具体功能名（如"取色""海报"），只写用户要达成的结果

---

### 第二步：问题优先 PRD（10分钟）

Marty Cagan 的方法：**先写问题，再写方案。**

**AI 参考输出格式（问题陈述）：**

```
## 问题陈述

### 谁的用户
[来自xx-goal的目标用户]

### 他们要完成的任务
[来自第一步的JTBD]

### 现在的痛点
1. [痛点1] — [影响的严重程度]
2. [痛点2] — [影响的严重程度]
3. [痛点3] — [影响的严重程度]

### 如果不解决
[后果：会怎样？对用户/对业务]

### 成功标准
[用户能做到什么，就算问题被解决了]
```

**方案空间（先发散再收敛）：**

```
方案A：[描述] — 优点：[ ] 缺点：[ ] 
方案B：[描述] — 优点：[ ] 缺点：[ ]
方案C：[描述] — 优点：[ ] 缺点：[ ]
```

**判断标准（给人）：** 痛点至少 3 条，每条标了严重程度；方案列了 ≥ 3 个才做选择。

**AI 执行约束：** 列出至少 3 个方案再选择，不要跳过发散直接选第一个想到的方案；选定方案必须写明理由。

---

### 第三步：MoSCoW 功能裁剪（5分钟）

**MVP 不是把完整产品砍一半，是找到能验证核心假设的最小功能集。**

**AI 参考输出格式：**

```
### Must（必须有）— 没有这些功能，核心任务无法完成
1. [功能] — 验证假设：[哪个假设]
2. [功能] — 验证假设：[哪个假设]

### Should（应该有）— 重要但MVP可以先不做的
1. [功能] — 可以用[替代方式]临时解决

### Could（可以有）— 锦上添花，等数据说话
1. [功能] — 等验证[什么]后再考虑

### Won't（不会有）— 明确不做的
1. [功能] — 因为[原因]
```

**Must 的判断标准（给人）：**
- 没有这个功能，用户能完成任务吗？不能 → Must
- 没有这个功能，能验证核心假设吗？不能 → Must
- 没有这个功能，只是体验差一点？→ Should 或 Could

**AI 执行约束：**
- Must 功能超过 3 个 → 必须重新聚焦假设，不能直接堆 5 个
- 每个 Must 功能必须对应一个要验证的假设，不对应说明功能多余
- Won't 必须写明原因，不能空着

**小象取色示例：**

| 分类 | 功能 | 说明 |
|------|------|------|
| Must | 摄像头实时取色 | 核心任务，没它产品不成立 |
| Must | AI 中式颜色命名 | 验证"用户更想要文化感命名"假设 |
| Should | AI 配色方案 | 可用预设色卡临时替代 |
| Could | 配色海报模板 | 等命名功能验证后再做 |
| Won't | 社区分享 | MVP 不做社交，避免分散验证焦点 |

---

### 第四步：最危险假设验证（5分钟）

从 xx-business 的风险分析继续：你的PRD是基于哪些假设设计的？

**AI 参考输出格式：**

```
### 核心假设
1. [用户假设]：用户真的有这个需求
   验证：[方法] 标准：[数字]
2. [行为假设]：用户会按预期的方式使用
   验证：[方法] 标准：[数字]
3. [价值假设]：用户觉得这比现有方案好
   验证：[方法] 标准：[数字]
4. [付费假设]：用户愿意为此付费
   验证：[方法] 标准：[数字]

### 最危险假设
[上面4个中，哪个错了产品就没有存在意义？]
验证方法：[最小成本方法]
验证时间：[天数]
```

**判断标准（给人）：** 最危险假设错了，产品就没有存在意义。如果找不出"错了就死"的假设，说明假设还没想透。

**AI 执行约束：** 4 类假设必须都写；最危险假设只能选 1 个；验证方法必须是"最小成本"，不能写"开发完整产品再验证"。

## AI 产品特殊考量

如果你的产品包含 AI 能力（大模型对话、图像处理等），PRD 需要额外包含：

**AI 参考输出格式（AI 补充模块）：**

```
### 模型选择
- 首选模型：[模型名] — 原因：[成本/效果/速度]
- 备选模型：[模型名] — 触发条件：[什么情况下切换]

### 质量标准
- 准确率要求：[最低可接受的准确率]
- 幻觉容忍度：[什么场景下幻觉不可接受 / 可接受]
- 响应时间：[用户可等待的最长时间]
- 失败降级：[模型不可用时怎么办？静默降级/提示重试/人工兜底]

### 成本控制
- 单次调用成本：[金额]
- 月度预算上限：[金额]
- 超预算策略：[降级方案/限流/付费墙]
```

**小象取色示例：**

| 维度 | 规格 |
|------|------|
| 首选模型 | 混元 hy3-preview（境内已备案、免费） |
| 备选模型 | 豆包（混元限流时切换） |
| 质量标准 | 命名准确率 ≥ 85%（按评估集，见 xx-data） |
| 幻觉容忍度 | 颜色名不可编造不存在的典故 |
| 响应时间 | 首字 ≤ 2s，全文 ≤ 8s |
| 失败降级 | 返回 hex + 通用色名（如"深红"）兜底 |
| 成本控制 | 单次 < ¥0.01，月预算 ¥500，超限触发限流 |

**AI 执行约束：** AI 补充模块不能省略"失败降级"——AI 一定会挂，必须有兜底。

## 输出：一页 PRD

**AI 参考输出格式（最终交付）：**

```markdown
## 一页 PRD — [产品名称]

### JTBD 任务
当 [场景] 时，我想要 [完成任务]，以便于 [达成目标]。
功能任务：[ ]
情感任务：[ ]
社会任务：[ ]

### 问题陈述
用户：[ ]
任务：[ ]
痛点：
1. [ ] — [严重程度]
2. [ ] — [严重程度]
3. [ ] — [严重程度]
不解决的后果：[ ]
成功标准：[ ]

### 方案选择
选定方案：[ ] 
理由：[ ]

### 功能规格
Must：
1. [功能] — 验证：[假设]
2. [功能] — 验证：[假设]
3. [功能] — 验证：[假设]
Should：[ ]
Could：[ ]
Won't：[ ]

### 最危险假设
[假设] → 验证方法：[ ] → 标准：[ ]

### AI 模块（如适用）
模型：[ ] 
质量标准：[ ]
降级策略：[ ]
成本控制：[ ]

### 度量指标
北极星：[ ]
验证指标：[ ]
```

**判断标准（给人，PRD 验收清单）：**
- [ ] JTBD 三层任务都写了，且没出现产品名/功能名
- [ ] 痛点至少 3 条，每条标了严重程度
- [ ] 方案空间列了 ≥ 3 个，选定方案有理由
- [ ] Must 功能 ≤ 3 个，每个对应一个假设
- [ ] Won't 写了原因
- [ ] 最危险假设只选了 1 个，且有最小成本验证方法
- [ ] AI 模块有失败降级策略
- [ ] 北极星指标可量化

**AI 执行约束：** 任一项未达标，PRD 不算完成，必须补齐再交付。

## 使用方式

把你的商业模式（xx-business的输出）或产品想法告诉我，我来引导你完成这份PRD。

**对话示例：**
- "我想做个能取色并起中式名字的小程序，帮我定义 MVP" → 我引导你走完四步，输出一页 PRD
- "这是我的 PRD 草稿，帮我看看有没有遗漏" → 我用验收清单逐项检查，重点看 AI 模块和假设
- "Must 功能我列了 6 个，是不是太多了" → 我帮你重新聚焦假设，砍到 ≤ 3 个
- "AI 命名挂了怎么办" → 我帮你补失败降级策略

## 与其他技能的关系

- **前置：** xx-goal 定目标、xx-business 理清商业模式（PRD 的输入）
- **下游：** xx-data 建 AI 评估集（PRD 里 AI 质量标准的落地）
- **下游：** xx-setup 项目初始化、xx-ai AI 能力接入（PRD 功能规格的落地）
- **回溯：** xx-track 上线后用真实数据验证 PRD 里的假设

## 方法论来源

- **Jobs-to-be-Done** — Clayton Christensen (哈佛商学院) / Tony Ulwick (Strategyn)
- **Outcome-Driven Innovation** — Tony Ulwick（86%创新成功率）
- **Problem-First PRD** — Marty Cagan，《Inspired: How to Create Tech Products Customers Love》
- **MoSCoW 优先级** — Must / Should / Could / Won't
- **Riskiest Assumption Test** — Lean Startup (Eric Ries)
